# 태스크 크기 가이드 — "30분 규칙"

> 이 문서는 모든 Sprint에 적용되는 태스크 크기 정의 기준입니다.

---

## 핵심 규칙

### 하나의 Sprint 태스크는 "30분 안에 결과를 눈으로 확인할 수 있는 크기"로 정의한다.

---

## 5가지 원칙

### 1. 시간 상한 (Max 30 Minutes)

- 하나의 태스크는 **최대 30분** 이내에 완료 가능해야 한다
- 30분을 초과하면 즉시 2~3개 하위 태스크로 분해한다
- Story Point 기준: 1 SP ≈ 15~30분

### 2. 시각적 검증 (Visual Verification)

- 완료 시 **눈으로 확인** 가능한 산출물이 있어야 한다
- "코드를 작성했다"가 아닌 "동작을 확인했다"가 완료 기준

| 산출물 유형 | 검증 방법 |
|-------------|-----------|
| API 엔드포인트 | curl/브라우저에서 응답 확인 |
| UI 컴포넌트 | 브라우저에서 렌더링 확인 |
| 비즈니스 로직 | 단위 테스트 통과 (터미널 로그) |
| 프롬프트 | 샘플 입력 → AI 출력 품질 확인 |
| 문서 | 마크다운 렌더링으로 내용 확인 |
| 설정/인프라 | 명령어 실행 → 성공 메시지 확인 |

### 3. 독립 실행 (Independent Execution)

- 다른 태스크 완료를 기다리지 않고 **단독으로** 실행·검증 가능
- 의존성이 있으면 **Mock/Stub**으로 격리
- 병렬 Agent 실행 시 파일 충돌이 없어야 함

### 4. 명확한 완료 기준 (Clear DoD)

- 각 태스크에 **Definition of Done**이 명시되어야 한다
- "~을 구현한다" ❌ → "~를 실행하면 ~가 표시된다" ✅

### 5. TDD 사이클 내 위치 (Red-Green-Refactor)

- **Red (2~5min):** 실패하는 테스트 또는 기대 결과를 먼저 정의
- **Green (10~20min):** 테스트를 통과시키는 최소 구현
- **Refactor (5~10min):** 코드 품질 개선
- 전체 사이클이 30분 이내

## 분해 기준

### 언제 태스크를 분해하는가?

```
태스크 예상 시간 > 30분?
  └── YES → 하위 태스크로 분해
       ├── 각 하위 태스크 ≤ 30분
       ├── 각 하위 태스크에 독립 검증 방법 있음
       └── 순서 의존성 최소화
  └── NO → 그대로 진행
```

### 분해 예시

```
❌ "파일 업로드 기능 구현" (2시간)
✅ 분해:
  ├── (20min) 파일 선택 UI + 미리보기 → 브라우저에서 파일 선택 확인
  ├── (20min) Drag & Drop 영역 → 파일 드롭 시 파일명 표시 확인
  ├── (25min) PDF 파싱 모듈 → 샘플 PDF 텍스트 추출 확인
  ├── (15min) DOCX 파싱 모듈 → 샘플 DOCX 텍스트 추출 확인
  └── (20min) 업로드 → 파싱 → 분석 통합 → 결과 표시 확인
```

```
❌ "AI 프롬프트 엔지니어링" (3시간)
✅ 분해:
  ├── (25min) 요약 프롬프트 v1 → 샘플 입력 → 요약 출력 품질 확인
  ├── (25min) 기능 목록 프롬프트 v1 → 기능 항목 추출 확인
  ├── (25min) 모호성 탐지 프롬프트 v1 → "빠르게" 탐지 확인
  ├── (25min) 테스트 포인트 프롬프트 v1 → TP 항목 생성 확인
  ├── (25min) QA 질문 프롬프트 v1 → 질문 생성 확인
  └── (20min) 누락 탐지 프롬프트 v1 → 누락 항목 생성 확인
```

## Agent 실행 시 적용

| 모델 | 태스크 유형 | 30분 규칙 적용 |
|------|-------------|----------------|
| **Opus** | 아키텍처, 프롬프트 설계 | 1 프롬프트 = 1 태스크 (검증: 샘플 출력 확인) |
| **Sonnet** | UI, API, 파서 구현 | 1 컴포넌트/모듈 = 1 태스크 (검증: 렌더링/테스트) |
| **병렬 Agent** | 독립 태스크 그룹 | 파일 충돌 없는 태스크만 병렬 실행 |
