## 한국어 요구사항 언어 패턴 분석

수집된 샘플에서 반복적으로 나타나는 서술 패턴:

### 주요 종결 패턴

| 패턴 | 예시 | 빈도 |
|---|---|---|
| `~해야 한다` | "포함해야 한다", "전달되어야 한다" | 높음 |
| `~할 수 있어야 한다` | "로그인할 수 있어야 한다", "조회할 수 있어야 한다" | 높음 |
| `~할 수 있다` | "추가, 수정, 삭제할 수 있다" | 중간 |
| `~를 지원한다` | "간편결제를 지원한다", "인앱 알림을 지원한다" | 중간 |
| `~를 표시한다` | "메시지를 표시한다", "알림 수를 표시한다" | 중간 |
| `~이 발송된다` | "이메일이 발송된다" | 낮음 |

### 주체 표현 방식

- **사용자**: 가장 빈번한 주체, 기능 요구사항에서 주로 사용
- **관리자**: 권한이 구분된 기능에서 명시
- **시스템**: 자동화 동작이나 제약 조건에서 사용
- **무주체 (암묵적 시스템)**: 정책 및 제한 조건 서술 시 자주 생략

## 분석 시사점

1. **모호성 탐지 필요성**: 전체 요구사항의 약 28%가 모호한 표현을 포함하여 AI 분석 도구의 모호성 탐지 기능이 실질적 가치를 가짐
2. **정량 기준 보완 필요**: 약 70%의 요구사항에 측정 가능한 기준이 없어 테스트 설계 시 기준 도출이 필요함
3. **한국어 패턴 일관성**: `~해야 한다` / `~할 수 있어야 한다` 패턴이 지배적이므로 프롬프트 설계 시 이 패턴을 기준으로 요구사항 파싱 로직을 구성할 수 있음
4. **도메인 다양성 확보**: 5개 샘플이 인증, 커머스, 콘텐츠, 알림, 관리 등 서로 다른 도메인을 커버하여 프롬프트의 범용성 검증에 적합함
