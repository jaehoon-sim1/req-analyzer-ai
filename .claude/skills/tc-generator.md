# TC Generator Skill

소프트웨어 요구사항(기획서/디스크립션)을 입력받아 구조화된 TestCase를 생성하는 스킬.
QA 엔지니어의 실무 TC 작성 표준을 따르며, JSON 형식으로 출력한다.

## 입력 형식

사용자가 제공하는 입력은 다음 중 하나:

1. **텍스트 입력**: 기능명 + 기능 상세 설명(디스크립션) + 주요 정책
2. **다중 페이지**: `[페이지명]` 구분자로 나뉜 여러 페이지의 디스크립션
3. **Figma 텍스트**: `[프레임: 프레임명]` 구분자로 나뉜 프레임별 텍스트

## TC 작성 규칙

### 핵심 원칙: 1건의 TC = 여러 절차 + 1개 기대결과

- 테스트 절차(procedure)는 여러 단계("1. ...", "2. ...", "3. ...")로 작성 가능
- **기대결과(expectedResult)는 반드시 1개만** 작성
- 확인할 결과가 여러 개면 각각 별도 TC로 분리
- 세부 항목이 있으면 하이픈(-)으로 나열 (번호 매기지 않음)

### Procedure (테스트 절차)

- 번호를 붙여 단계별 작성: "1. ~ 한다.\n2. ~ 한다.\n3. ~ 한다."
- 메뉴 경로는 대괄호: "1. [관리자 > 컨텐츠 관리 > 약품정보] 메뉴를 선택한다."
- UI 요소명은 작은따옴표: "1. '컨텐츠 유형' 영역의 구성을 확인한다."
- 버튼은 대괄호: "1. [적용] 버튼을 선택한다."
- 동사는 한국어 정중체: ~선택한다, ~확인한다, ~입력한다, ~클릭한다

### Expected Results (기대 결과)

- 1개의 기대결과만 작성 (번호 없이)
- 예: "약품정보 페이지가 노출된다."
- 예: "아래 구성이 제공된다.\n- '컨텐츠 유형' 타이틀 문구\n- [유형 선택] 버튼\n- [초기화] 버튼"
- 동사는 한국어 정중체: ~노출된다, ~제공된다, ~표시된다, ~반영된다, ~활성화된다
- 절대로 "1. ...", "2. ..." 형태로 여러 결과를 나열하지 않음

### Depth 분류 기준

| Depth | 역할 | 예시 |
|-------|------|------|
| depth1 | 대분류 기능 영역 | "컨텐츠 관리", "회원관리", "로그인" |
| depth2 | 중분류 하위 기능 | "약품정보", "메디컬트렌드" |
| depth3 | 소분류 세부 기능 (선택) | "컨텐츠 유형", "검색 필터" |
| depth4 | 세부 팝업/모달 (선택) | "컨텐츠 유형 선택" |
| depth5 | 최세부 분류 (선택) | - |

### Precondition (사전 조건)

해당하는 경우에만 간결하게 작성. 대부분의 TC에는 불필요.
예: "임의 컨텐츠 유형이 선택된 상태", "배포전 등록된 게시글인 경우"

### TC No. 형식

`{접두어}_{3자리 번호}` 형식. 예: `Admin_001`, `Login_002`, `Board_003`

## 섹션 구성 가이드

각 섹션은 기능의 주요 영역별로 구분:

- **화면 구성**: UI 요소 존재 여부, 초기 상태 확인
- **정상 동작**: 정상적인 기능 수행 케이스
- **유효성 검증**: 입력값 검증, 필수값 미입력, 잘못된 입력
- **정책 검증**: 주요 정책에 명시된 비즈니스 규칙 검증
- **경계값/예외**: 특수문자, 최대길이, 연속 동작 등

## JSON 출력 형식

```json
{
  "sections": [
    {
      "storyId": "Admin",
      "sectionTitle": "관리자페이지 > 컨텐츠 관리 > 약품정보",
      "tcPrefix": "Admin",
      "testCases": [
        {
          "depth1": "컨텐츠 관리",
          "depth2": "약품정보",
          "depth3": "컨텐츠 유형",
          "testType": "화면 구성",
          "precondition": "",
          "procedure": "1. [관리자 > 컨텐츠 관리 > 약품정보] 메뉴를 선택한다.\n2. '컨텐츠 유형' 영역을 확인한다.",
          "expectedResult": "아래 구성이 제공된다.\n- '컨텐츠 유형' 타이틀 문구\n- [유형 선택] 버튼\n- [초기화] 버튼",
          "docInfo": "CNT_DRG_01"
        }
      ]
    }
  ]
}
```

## 다중 페이지 처리

입력이 `[페이지명]` 구분자로 나뉜 경우:

- 페이지 순서에 관계없이 전체 맥락을 파악
- 논리적 흐름(환경설정 → 화면 구성 → 기능 동작 → 예외 처리)에 맞게 TC 정리
- 각 페이지의 내용을 누락 없이 반영하되, 중복 내용은 하나로 통합

## 응답 제약

- JSON 문자열 안에서 줄바꿈은 반드시 `\n` (이스케이프) 사용
- 전체 JSON이 반드시 완전한 형태로 끝나야 함
- TC가 너무 많아지면 핵심 TC만 선별
- procedure와 expectedResult는 간결하게 작성

## 사용 예시

**입력:**
```
기능명: 로그인
기능 상세 설명: 아이디와 비밀번호를 입력하여 로그인하는 화면. [로그인] 버튼, [아이디 찾기]/[비밀번호 찾기] 링크 제공.
주요 정책: 비밀번호 5회 연속 오류 시 계정 잠금 (30분)
```

**출력 (TC 예시):**
```
TC No: Login_001
Depth: 로그인 > 화면 구성
절차: 1. 로그인 페이지에 접속한다.
기대결과: 아이디 입력 필드, 비밀번호 입력 필드, [로그인] 버튼이 노출된다.

TC No: Login_002
Depth: 로그인 > 인증 실패
절차: 1. 잘못된 비밀번호를 입력한다.\n2. [로그인] 버튼을 클릭한다.
기대결과: '아이디 또는 비밀번호가 일치하지 않습니다.' 오류 문구가 노출된다.

TC No: Login_003
Depth: 로그인 > 계정 잠금
절차: 1. 비밀번호를 5회 연속 잘못 입력한다.
기대결과: '계정이 잠겼습니다. 30분 후 다시 시도해주세요.' 문구가 노출된다.
```
